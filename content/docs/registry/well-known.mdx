## `/.well-known/commerce-gateway.json`

Any domain running a Commerce Gateway can publish discovery metadata at:

```
https://{domain}/.well-known/commerce-gateway.json
```

This lets clients discover your gateway **without** querying a central registry first. It is part of the Commerce Registry Protocol; field requirements match [spec/well-known.md](https://github.com/betterdata/bd-forge-main/blob/main/commerce-registry-protocol/spec/well-known.md).

## Schema (v1.0)

```json
{
  "version": "1.0",
  "brand": "Example Brand",
  "gateway": {
    "endpoint": "https://api.example.com/llm/v1",
    "protocol": "mcp",
    "capabilities": {
      "catalog_search": true,
      "pricing": "public",
      "inventory": "real_time",
      "checkout": false
    },
    "auth": {
      "type": "none"
    }
  },
  "gtins": {
    "prefixes": ["0012345", "0012346"]
  },
  "categories": ["beauty.makeup", "beauty.skincare"],
  "contact": "commerce@example.com"
}
```

Required fields include `version`, `brand`, `gateway.endpoint`, `gateway.protocol`, and the four capability keys under `gateway.capabilities`. Optional fields include `gateway.auth`, `gtins.prefixes`, `categories`, and `contact`.

## Serving the file

**Next.js (App Router)** — Add `public/.well-known/commerce-gateway.json`. Files under `public/` are served at the root path.

**Express**

```js
app.get("/.well-known/commerce-gateway.json", (req, res) => {
  res.json({
    version: "1.0",
    brand: "Your Brand",
    gateway: {
      endpoint: "https://api.yourdomain.com/mcp",
      protocol: "mcp",
      capabilities: {
        catalog_search: true,
        pricing: "public",
        inventory: "real_time",
        checkout: false,
      },
    },
  });
});
```

## Registry fallback

Resolution implementations may try **well-known discovery** on candidate domains, then fall back to **`GET /api/resolve/brand`** (and related endpoints) on the configured registry. Both paths are defined in the protocol.
