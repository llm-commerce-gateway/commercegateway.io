## Resolution flow

1. **Intent** — The user or agent mentions a brand, GTIN, or category (for example `@shop Nike` or a barcode).
2. **Registry query** — The client calls the registry HTTP API (`/api/resolve/brand`, `/api/resolve/gtin/{gtin}`, or `/api/resolve/category/...`) or uses tooling built on [`@betterdata/commerce-gateway`](https://www.npmjs.com/package/@betterdata/commerce-gateway) (for example `createRegistryClient` / `RegistryClient`).
3. **Endpoint** — The registry returns a **gateway** object: `endpoint`, `protocol` (often `mcp`), and **capabilities** (catalog search, pricing visibility, inventory mode, checkout).
4. **Direct gateway call** — The LLM runtime connects to that endpoint and invokes commerce tools there. The registry is not in the hot path for catalog reads.

## Registration flow

1. **Register** — `POST /api/gateways` with brand, domain, public endpoint, protocol, and capabilities.
2. **Verify** — Prove domain control via DNS TXT on `_commerce-gateway.{domain}` or a token at `/.well-known/commerce-gateway-verify.json`, then `POST /api/gateways/{id}/verify`.
3. **Optional GTINs** — `POST /api/gateways/{id}/gtins` to associate product identifiers with your gateway for GTIN resolution.

## Fallback without a central registry

If a domain publishes **`/.well-known/commerce-gateway.json`**, clients can discover the gateway without hitting a central registry first. Resolution logic may still fall back between registry matches and well-known discovery; see the [resolution spec](https://github.com/betterdata/bd-forge-main/blob/main/commerce-registry-protocol/spec/resolution.md).
